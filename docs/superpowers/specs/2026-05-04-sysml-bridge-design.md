# SysML Bridge — Design Spec

> Claude Code skill suite + MCP server for bidirectional natural-language ↔ SysML v2 MBSE workflows, portable from open-source dev to Cameo Enterprise Architecture.

## Problem

MBSE workflows in Cameo are manual, tedious, and locked behind an expensive license. SysML v2's textual notation makes LLM-driven model generation viable for the first time — but there's no tooling that bridges Claude Code into the SysML v2 ecosystem. This project builds that bridge.

## Architecture

Three layers with strict boundaries:

```
┌─────────────────────────────────────────────┐
│           NATURAL LANGUAGE LAYER            │
│   Claude Code + skill suite (user works here)│
└────────────────────┬────────────────────────┘
                     │ MCP tools
┌────────────────────▼────────────────────────┐
│              BRIDGE LAYER                   │
│   MCP server → SMAPS REST API (Docker)      │
│   stores elements, validates, queries model │
└────────────────────┬────────────────────────┘
                     │ SysML v2 files + API
┌────────────────────▼────────────────────────┐
│              MODEL LAYER                    │
│   .sysml text files + SMAPS model store     │
│   Eclipse plugin for visual diagrams        │
│   → swap endpoint to Cameo at work          │
└─────────────────────────────────────────────┘
```

**Key principle:** The MCP server is the only thing that knows where the model lives. Skills never talk directly to files or the API — they go through the MCP server. Cameo portability becomes a configuration change, not a rewrite.

**Session state:** A lightweight `.mbse-session.json` file tracks where the project is in the MBSE lifecycle — current phase, what's been generated, what's pending. Skills read it to know context so the workflow feels stateful, not amnesiac.

## Dev Stack

| Purpose | Tool |
|---|---|
| Skills / AI layer | Claude Code + skill suite (installed as plugin) |
| Model API | SMAPS server via SysML v2 Pilot Implementation (Docker) |
| File editing | VS Code + SysIDE extension |
| Visual diagrams | Eclipse SysML v2 plugin (free) |
| Diagram rendering in-flow | Mermaid / PlantUML generated by skills |
| Portability target | Cameo Enterprise Architecture (same SMAPS API) |

## Skill Suite

11 skills mapping to the full MBSE lifecycle:

### Core Workflow Skills

| Skill | What it does |
|---|---|
| `/mbse-init` | Bootstrap a new project — stakeholder needs, system context, CONOPS sketch. Takes natural language system description, outputs `.sysml` package structure, `.mbse-session.json`, initial stakeholder needs and use cases. |
| `/mbse-requirements` | Generate or refine requirements — IDs, parent-child hierarchy, verifiability assessment. Derives from stakeholder needs + CONOPS. Phase-aware: starts fresh during inception, refines during later phases. |
| `/mbse-build` | The construction layer. Build specific SysML v2 artifacts from requirements via subcommands: |
| | `mbse-build bdd` — part definitions, generalizations, compositions |
| | `mbse-build ibd` — part usages, port definitions, connections, flows |
| | `mbse-build activity` — action definitions, control/data flows |
| | `mbse-build sequence` — interaction usages, message ordering |
| | `mbse-build state` — state definitions, transitions |
| | `mbse-build parametric` — constraint definitions, analysis bindings |

### Analysis & Verification Skills

| Skill | What it does |
|---|---|
| `/mbse-trace` | Build and update traceability links — requirements → model elements → verification. Generates satisfy, verify, refine, allocate relationships + summary matrix. |
| `/mbse-validate` | Check model completeness, consistency, orphaned elements, coverage gaps. Reconciles session state against SMAPS as first step. |
| `/mbse-verify` | V&V planning — map each requirement to a verification method (Test / Analysis / Inspection / Demonstration). Generates verification matrix. |
| `/mbse-trade` | Weighted trade studies — Pugh matrices, MOE/MOP scoring, decision rationale with traceability to requirements. |
| `/mbse-kpp` | Define and track Key Performance Parameters, Measures of Effectiveness, Measures of Performance — quantitative measures with thresholds and current values. |

### Utility Skills

| Skill | What it does |
|---|---|
| `/mbse-views` | Generate SysML v2 viewpoints/views — stakeholder-specific slices of the model (operator view, maintainer view, program manager view). |
| `/mbse-diagram` | Render current model structure as Mermaid or PlantUML. Supports BDD-style, IBD-style, sequence, activity, state diagrams. Takes element ID or scope. |
| `/mbse-query` | Ask natural language questions about the model. Returns NL answers grounded in actual model elements. |

### Workflow Flow

```
init → conops → requirements → kpp
                     │
          ┌──────────┼──────────────┐
          ▼          ▼              ▼
     build bdd   build activity   build state
     build ibd   build sequence   build parametric
          │          │              │
          └──────────┼──────────────┘
                     ▼
              trade (at any point)
                     ▼
               trace → verify → validate
                     ▼
                   views
                     ▼
                  diagram (at any point)
                  query  (at any point)
```

Non-linear — build what you need, trace and validate at any point, go back and refine.

### Session State File

```json
{
  "project": "ANGARS-v2",
  "phase": "architecture",
  "smaps_endpoint": "http://localhost:9000",
  "history": [
    { "skill": "mbse-init", "timestamp": "...", "elements_created": 12 },
    { "skill": "mbse-requirements", "timestamp": "...", "elements_created": 47 }
  ],
  "pending": ["trace requirements to blocks", "validate coverage"]
}
```

Skills read this to know context. `/mbse-requirements` during the `architecture` phase knows it's refining, not starting fresh.

## MCP Server Design

**Server name:** `sysml-bridge`

**Configuration:**

```json
{
  "SMAPS_ENDPOINT": "http://localhost:9000",
  "PROJECT_ID": "angars-v2"
}
```

Change `SMAPS_ENDPOINT` to Cameo's server at work — everything else stays the same.

### MCP Tools

| Tool | Purpose | Maps to SMAPS |
|---|---|---|
| `create_element` | Create any SysML v2 element (part def, requirement, action, etc.) | `POST /projects/{id}/commits/{id}/elements` |
| `get_element` | Retrieve element by ID | `GET /projects/{id}/commits/{id}/elements/{id}` |
| `query_elements` | Find elements by type, name, relationship | `GET /projects/{id}/commits/{id}/elements?filter=...` |
| `update_element` | Modify an existing element | `PATCH /projects/{id}/commits/{id}/elements/{id}` |
| `delete_element` | Remove an element | `DELETE ...` |
| `create_relationship` | Link elements — satisfy, verify, refine, allocate, compose | `POST ...` with relationship type |
| `query_relationships` | Get all relationships for an element or type | `GET ...` filtered by relationship kind |
| `validate_model` | Run consistency/completeness checks | Custom logic over SMAPS queries |
| `get_traceability` | Walk the relationship graph for an element | Multi-hop query over relationships |
| `export_sysml` | Export model or scope as `.sysml` text | Serialize elements to SysML v2 textual notation |
| `import_sysml` | Parse `.sysml` text and push to SMAPS | Parse + `POST` elements |
| `get_project_state` | Return summary — element counts by type, coverage stats | Aggregation query |

**Design decisions:**
- **Tools are element-agnostic.** `create_element` handles part definitions, requirements, actions, states — the `type` parameter determines what gets created. Skills know which types to use; the MCP server doesn't need SysML domain logic.
- **Relationships are first-class.** Traceability is the entire point of MBSE. `create_relationship` and `query_relationships` are separate tools, not buried inside element creation.
- **`export_sysml` / `import_sysml` enable the file-based workflow.** Edit in VS Code with SysIDE, re-import. SMAPS store is source of truth, files are interchange.

## Error Handling

**SMAPS server not running:**
- Every MCP tool checks connectivity on first call
- Clear error: "SMAPS server not reachable at http://localhost:9000 — run `docker compose up` in the sysml-bridge directory"
- Session file still tracks intent — when the server comes back, no work is lost

**Invalid SysML v2 syntax on import:**
- `import_sysml` runs the parser before pushing to SMAPS
- Returns line-by-line errors with the specific syntax violation
- Skill layer presents these as actionable fixes, not raw parser output

**Model conflicts during bidirectional flow:**
- Editing a `.sysml` file in VS Code then calling a skill that modifies the same element
- MCP server uses SMAPS commit model — each skill operation is an atomic commit
- On conflict: skill reports what changed since last sync, asks whether to merge or override

**Session state drift:**
- `.mbse-session.json` gets out of sync with the actual model (manual edits, crashed skill)
- `/mbse-validate` always reconciles session state against SMAPS as its first step
- Any skill can call `get_project_state` to rebuild the session from truth

**Cameo endpoint differences:**
- SMAPS spec is standardized but implementations may differ in edge cases
- `smaps-client.ts` is the single place to handle vendor quirks
- Integration tests run against both Pilot and Cameo endpoints (when available)

## Testing Strategy

| Layer | Approach |
|---|---|
| MCP server tools | Unit tests with mocked SMAPS responses |
| SMAPS client | Integration tests against Pilot in Docker (CI-friendly) |
| SysML parser/serializer | Unit tests with known `.sysml` fixtures |
| Skills | Manual testing through Claude Code + the ANGARS example project |
| End-to-end | Script that runs init → requirements → build → trace → validate against a live Pilot instance |

## Portfolio Story

The ANGARS example project is the demo: rebuild your JHU capstone through the tool, proving it handles a real 189-requirement, 6-subsystem, 10-KPP system. The portfolio page shows the same system engineered two ways — manually in Cameo, and through Claude Code with sysml-bridge.
